## Key Learnings

- **Project:** wheels-app
- **vehicle_models table**: עמודת `make_he` היא NOT NULL. בעת הוספת רכב ב-API route, אם `make_he` לא נשלח, חייבים לתרגם אוטומטית מ-English → Hebrew.
- **translateMakeToHebrew**: הפונקציה ב-`/api/vehicle-models/route.ts` מחפשת קודם ב-`englishToHebrewMakes` (היפוך של `hebrewToEnglishMakes`), אחר כך Google Translate API עם `GOOGLE_VISION_API_KEY`, ואז פולבק לשם האנגלי.
- **handleAddScrapedResult** בדף admin/vehicles: שולח `make_he: plateVehicleInfo?.manufacturer_he || ''` — כשב-manual mode `plateVehicleInfo` הוא null ולכן `make_he` ריק. הפתרון הוא בצד שרת, לא ב-client.
- The `login_log` table (src/lib/login-log.ts) exists in the live Supabase DB but has NO migration file in supabase/migrations — it was created directly in the dashboard. Its `user_id` column is a FK to `users(id)` with no `ON DELETE CASCADE`, unlike `user_roles`/`webauthn_*` which do cascade (see supabase/migrations/20260405_unified_users.sql, 20260424_add_webauthn.sql). Any raw `supabase.from('users').delete()` will 500 with a FK violation for any user who ever logged in, unless `login_log.user_id` is nulled first.
- login_log rows already store full_name/phone independently, so nulling user_id (not deleting the row) is safe — login-report and analytics routes fall back to `full_name` when `user_id` is null.
- No Supabase DB credentials are available in this coding session — new migration `.sql` files under `supabase/migrations/` are never auto-applied. Always tell the user explicitly to run them via the Supabase SQL editor or `supabase db push` after writing one.
- SQL reserved words used as column names (e.g. `offset`, which collides with `LIMIT...OFFSET`) must be double-quoted in migration files (`ADD COLUMN "offset" NUMERIC`), but Supabase-js/PostgREST calls (`.insert({offset: 5})`) are unaffected — no quoting needed in application code.
- `src/app/admin/vehicles/page.tsx` renders its per-row "⋮" action menu (edit/delete/etc.) in two separate places in the JSX: once for the mobile card view (~line 1618) and once for the desktop table view (~line 1735+, shifts with edits). Any new per-row action must be added to both or it'll silently only work on one viewport.
- `wheels.tire_size` (free text, e.g. "205/55") and the vehicle lookup's `vehicle.front_tire` (full "WWW/AARdd" incl. rim) are NOT the same format — `wheels.tire_size` deliberately omits the rim suffix since rim size is captured separately in `wheels.rim_size`. `getTireDiameterMm()` in src/lib/vehicle-mappings.ts accounts for this via an optional rim-size override parameter. Reused by both src/app/search/page.tsx and src/app/stations/page.tsx, which duplicate the same per-wheel result-card rendering logic (rimMismatch/cbNeedsRing/diameterMismatch) — keep both in sync when touching one.
- **The live Supabase schema drifts from supabase/migrations/** — confirmed twice in one session (login_log had no migration file at all; user_roles_user_id_fkey's ON DELETE CASCADE in 20260405_unified_users.sql doesn't match the live constraint, which was NO ACTION). Don't trust migration file contents as ground truth for constraint/cascade behavior when debugging FK errors — ask the user to query `information_schema.referential_constraints` directly (query saved in Do-Not-Repeat below), and prefer application-level explicit deletes of dependent rows over relying on ON DELETE CASCADE for any table touching `users`.
- [FIXED 2026-07-02] `src/lib/version.ts` VERSION used to be auto-bumped by a local-only `.git/hooks/pre-commit` (untracked, machine-local) with a string-replace bug (`m.replace(z, +z+1)` replaced the first textual occurrence of the patch digit(s) anywhere in "VERSION = 'X.Y.Z'", not the patch position — so when major/minor/patch shared a digit, e.g. `4.4.4`, it silently bumped the wrong segment). Fixed by removing that hook and wiring the correct, already-existing `scripts/bump-version.js` (proper 0-9 rollover) through husky instead: `"prepare": "husky"` in package.json + `.husky/pre-commit` runs `node scripts/bump-version.js && git add src/lib/version.ts`. This is now tracked in git, so it runs on every clone after `npm install` (unlike the old raw git hook). Verified working end-to-end via a real commit (18.0.3 → 18.0.4).
- [2026-07-02] A GitHub PR merge + `git pull origin main` + branch checkout happened concurrently while a coding session had uncommitted work on `claude/universal-wheel-integration-sojrdl` — git auto-stashed the uncommitted work (twice, under different messages) before switching branches, rather than losing it. Lesson: if files unexpectedly appear reverted mid-session with no error, check `git stash list` and `git reflog` before assuming work is lost — it's very likely sitting in a stash. `git stash apply` can still fail on unrelated bystander files (e.g. `.wolf/*` auto-log files touched by hooks after the stash was created); resolve those via `git checkout -- <file>` immediately before retrying the apply (hooks may re-dirty them between attempts, so chain checkout+apply in one shell call).

## Do-Not-Repeat

- [2026-07-02] Deleting a user via `supabase.from('users').delete().eq('id', userId)` fails with a 500 ("שגיאה פנימית") whenever the user has login_log rows, because that FK has no CASCADE (see Key Learnings). This also silently broke `/api/admin/users/merge`: it moves roles to keep_id FIRST, then deletes delete_id — when the delete step hit this same FK error, roles had already moved but delete_id survived as a roleless ghost account, which read to the user as "role duplication". Fix: always `update({ user_id: null }).eq('user_id', <id>)` on `login_log` before deleting a row from `users`, in any endpoint that deletes users (both `[userId]/route.ts` and `merge/route.ts` now do this). A DB-level fix (ON DELETE SET NULL on the FK) was added in supabase/migrations/20260702_fix_login_log_user_fk.sql — user confirmed it was applied via Supabase SQL editor.
- [2026-07-02] After fixing login_log, user delete STILL 500'd. User ran a manual DELETE in the Supabase table editor and got the real Postgres error verbatim: `user_roles_user_id_fkey` also has no ON DELETE CASCADE in the live DB, despite 20260405_unified_users.sql defining it that way. Confirmed by querying `information_schema.referential_constraints` — user_roles was `NO ACTION`, while webauthn_credentials/webauthn_challenges were correctly `CASCADE` and login_log was `SET NULL` post-fix. **General lesson: this project's live Supabase schema does not reliably match the migration files in supabase/migrations/ — treat migration file contents (including ON DELETE behavior) as aspirational, not ground truth, when debugging FK errors.** When a delete/cascade bug is suspected, ask the user to run this against `information_schema` rather than trusting the migration files: `SELECT tc.table_name, tc.constraint_name, rc.delete_rule FROM information_schema.table_constraints tc JOIN information_schema.referential_constraints rc ON tc.constraint_name = rc.constraint_name JOIN information_schema.key_column_usage kcu ON kcu.constraint_name = tc.constraint_name WHERE tc.constraint_type = 'FOREIGN KEY' AND EXISTS (SELECT 1 FROM information_schema.constraint_column_usage ccu WHERE ccu.constraint_name = tc.constraint_name AND ccu.table_name = '<target_table>');`. Fix: app code now deletes `user_roles`/`webauthn_credentials`/`webauthn_challenges` explicitly before deleting a user instead of trusting any FK's delete behavior (src/app/api/admin/users/[userId]/route.ts, src/app/api/admin/users/merge/route.ts). DB-level fix in supabase/migrations/20260702_fix_user_roles_user_fk.sql — user confirmed applied ("Success. No rows returned").
